## 一、总体策略

### 1.1 分阶段实施原则

基于系统复杂度，采用**MVP先行 + 迭代扩展**策略：

```
Phase 1 (3个月)     Phase 2 (3个月)     Phase 3 (3个月)     Phase 4 (3个月)
┌─────────────┐   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│ MVP核心数据层│ → │ AI能力验证  │ → │ 应用层完善  │ → │ 全场景覆盖  │
│ + 单ERP接入 │   │ + POC      │   │ + 多ERP    │   │ + CRM集成  │
└─────────────┘   └─────────────┘   └─────────────┘   └─────────────┘
```

### 1.2 MVP范围

**Phase 1 只做最小可用系统**：

| 模块 | MVP范围 | 产出 |
|------|---------|------|
| 数据接入 | 金蝶ERP + 1个银行 | 数据管道打通 |
| 数据湖 | Iceberg + Doris | 原始层+标准层上线 |
| 数据集 | AR应收主题 | 第一个数据集 |
| AI层 | 本地LLM + RAG基础 | 自然语言查询Poc |
| 应用 | 飞书机器人 + 1个看板 | 可演示Demo |

**POC验收标准**：

```
NLQuery:
□ "本月应收总额是多少" → 返回正确数字
□ "哪些客户逾期了" → 返回逾期客户列表
□ 响应时间 < 5秒

□ 逾期率异动 → 给出Top3原因
□ 收入下降 → 给出维度拆解
```

## 四、技术债务与风险

### 4.1 主要风险

| 风险 | 影响 |  Mitigation |
|------|------|-------------|
| 金蝶API不稳定 | 数据延迟 | 降级到数据库直连 |
| LLM推理慢 | 用户体验差 | 缓存 + 降级到规则 |
| 多租户性能 | 查询延迟 | 物化视图 + 预聚合 |
| 数据质量差 | AI输出不准 | 加强质控 + 人工确认 |

### 4.2 技术债务

| 债务项 | 处理时机 | 说明 |
|--------|----------|------|
| CRM冲突解决算法 | Phase 4 | 暂用规则简化版 |
| 列级血缘追踪 | Phase 3后 | MVP先用表级血缘 |
| 数据脱敏完善 | Phase 2后 | 先用列级脱敏 |
